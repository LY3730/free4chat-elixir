# Free4Chat Docker 构建探索记录

## 背景

Free4Chat 是一个基于 WebRTC 的视频聊天应用，后端使用 Elixir/Phoenix + Membrane RTC Engine。尝试在 Windows 本地使用 Docker 构建并运行后端服务时遇到了一系列网络和环境问题。以下是完整的排查和解决过程。

---

## 问题一：Docker 无法拉取基础镜像

### 错误现象

```bash
docker build --build-arg DASHBOARD_AUTH_USERNAME=admin --build-arg DASHBOARD_AUTH_PASSWORD=password -t free4chat .
```

```
ERROR: failed to solve: alpine:3.16: failed to resolve source metadata for docker.io/library/alpine:3.16:
unable to fetch descriptor ... which reports content size of zero: invalid argument
```

### 原因分析

Dockerfile 使用了两个基础镜像：
- `elixir:1.14.1-alpine`（第 1 行，构建阶段）
- `alpine:3.16`（第 51 行，运行阶段）

由于国内网络限制，Docker Hub (`docker.io`) 访问不稳定，导致无法拉取镜像元数据。

### 解决方案

在 Docker Desktop → Settings → Docker Engine 中配置镜像加速器：

```json
{
  "registry-mirrors": [
    "https://docker.1ms.run",
    "https://docker.xuanyuan.me"
  ]
}
```

点击 Apply & Restart 后重新构建，镜像拉取成功。

## 问题二：容器内无法访问 GitHub（Git 依赖拉取失败）

### 错误现象

镜像拉取成功后，构建进行到 `mix setup` 步骤时报错：

```
ERROR [build 9/11] RUN mix setup
* Getting membrane_rtc_engine (https://github.com/membraneframework/membrane_rtc_engine.git)
fatal: unable to access 'https://github.com/membraneframework/membrane_rtc_engine.git/':
Failed to connect to github.com port 443 after 31 ms: Couldn't connect to server
```

### 原因分析

查看 `mix.lock` 文件发现 `membrane_rtc_engine` 使用的是 **Git 仓库源**而非 Hex 包：

```elixir
# mix.lock 第 30 行
"membrane_rtc_engine": {:git, "https://github.com/membraneframework/membrane_rtc_engine.git", ...}
```

`mix setup` 执行时会调用 `git clone` 从 GitHub 拉取该仓库，但 Docker 容器内网络无法直接访问 GitHub。

### 尝试的解决方案

#### 方案 A：GitHub 代理（失败）

在 Dockerfile 中添加 Git URL 重写规则：

```dockerfile
RUN git config --global url."https://ghproxy.net/https://github.com/".insteadOf "https://github.com/"
```

结果：代理 `ghproxy.net` 不稳定，报 `PROTOCOL_ERROR`。

换用 `mirror.ghproxy.com` 仍然不稳定。

#### 方案 B：本地克隆 + Path 依赖（成功）

**核心思路**：提前在本地把 GitHub 仓库下载好，Docker 构建时直接从本地读取，完全绕过网络问题。

具体步骤：

**1. 本地克隆仓库**

```bash
cd backend
git clone https://github.com/membraneframework/membrane_rtc_engine.git deps/membrane_rtc_engine
```

**2. 修改 `mix.exs`**

```elixir
# 修改前（第 43 行）
{:membrane_rtc_engine, github: "membraneframework/membrane_rtc_engine", override: true},

# 修改后
{:membrane_rtc_engine, path: "deps/membrane_rtc_engine", override: true},
```

将 Git 远程依赖改为本地路径依赖。

**3. 修改 `Dockerfile`**

移除 Git 代理配置行，添加 COPY 指令将本地仓库复制进容器：

```dockerfile
COPY lib lib
# copy local membrane_rtc_engine (pre-cloned from GitHub, no network needed)
COPY deps/membrane_rtc_engine deps/membrane_rtc_engine

RUN mix setup
```

**4. 修改 `.dockerignore`**

`.dockerignore` 默认排除了 `/deps/` 目录，需要放行 `membrane_rtc_engine`：

```
/deps/
!/deps/membrane_rtc_engine
```

**5. 重新构建 — 成功！**

```
[+] Building 95.9s (23/23) FINISHED
 => naming to docker.io/library/free4chat:latest
```

全部 23 个步骤顺利完成。

## 问题三：运行容器缺少必要环境变量

### 错误现象

```bash
docker run -p 50000-50050:50000-50050/udp -p 4000:4000/tcp \
  -e INTEGRATED_TURN_PORT_RANGE=50000-50050 \
  -e INTEGRATED_TURN_IP=127.0.0.1 \
  free4chat
```

```
ERROR! Config provider Config.Reader failed with:
(System.EnvError) could not fetch environment variable "VIRTUAL_HOST" because it is not set
```

### 原因分析

Phoenix 应用的 `runtime.exs` 在启动时通过 `System.fetch_env!/1` 强制要求多个环境变量必须设置。

查看 `.env` 文件确认需要的变量：

```
DASHBOARD_AUTH_USERNAME=xxx
DASHBOARD_AUTH_PASSWORD=xxx
VIRTUAL_HOST=localhost
USE_TLS=false
SECRET_KEY_BASE={mix phx.gen.secret}
INTEGRATED_TURN_IP=100.83.1.67
```

### 解决方案

运行容器时传入所有必需的环境变量：

```bash
docker run -d --name free4chat \
  -p 4000:4000/tcp \
  -p 50000-50050:50000-50050/udp \
  -e DASHBOARD_AUTH_USERNAME=admin \
  -e DASHBOARD_AUTH_PASSWORD=password \
  -e VIRTUAL_HOST=localhost \
  -e USE_TLS=false \
  -e SECRET_KEY_BASE=free4chat_secret_key_base_2024 \
  -e INTEGRATED_TURN_IP=127.0.0.1 \
  -e INTEGRATED_TURN_PORT_RANGE=50000-50050 \
  free4chat
```

容器成功启动并正常运行。

## 总结

| 问题 | 根原因 | 解决方法 |
|------|--------|----------|
| Docker 无法拉取镜像 | Docker Hub 国内访问受限 | 配置 registry-mirrors 加速器 |
| mix setup 拉取 GitHub 依赖失败 | membrane_rtc_engine 是 Git 依赖，容器内无网络 | 本地预克隆 + 改为 path 依赖 + COPY 进容器 |
| 容器启动报错缺少环境变量 | runtime.exs 强制要求多个 ENV VAR | docker run 时传入所有必需变量 |

## 最终修改的文件清单

| 文件 | 修改内容 |
|------|----------|
| `backend/mix.exs` | `membrane_rtc_engine` 从 github 改为 path 依赖 |
| `backend/Dockerfile` | 移除 git 代理行，添加 COPY deps/membrane_rtc_engine |
| `backend/.dockerignore` | 添加 `!/deps/membrane_rtc_engine` 放行规则 |
| `backend/deps/membrane_rtc_engine/` | 新增，从 GitHub 克隆的本地副本 |

## 经验教训

1. **Elixir Mix 项目的 Git 依赖**在 Docker 构建中是常见痛点，尤其是在网络受限环境中。`path` 依赖是可靠的替代方案。
2. **`.dockerignore` 的 `/deps/` 排除规则**会影响到手动 COPY 的依赖目录，需要注意放行。
3. **Phoenix 生产环境**通过 `runtime.exs` 和 `System.fetch_env!/1` 严格管理配置，所有必需的环境变量必须在容器启动时提供。
